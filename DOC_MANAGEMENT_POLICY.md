# 文档开发管理规范

**版本**: v1.0  
**生效日期**: 2026-02-25  
**适用范围**: TO-DO App 项目全体文档编写人员

---

## 📜 总则

### 核心原则

1. **代码为准**: 文档必须无条件适配代码，代码已封板时不可修改代码来适配文档
2. **及时同步**: 代码变更后必须立即更新相关文档
3. **准确完整**: 文档内容必须准确反映实际功能，不得遗漏或夸大
4. **易于维护**: 文档结构清晰，便于查找和更新

---

## 📁 文档分类体系

### 1. 核心文档（Core Documents）

**必须文档**:
- `README.md` - 项目概览、功能列表、安装指南
- `CHANGELOG.md` - 版本更新日志
- `LICENSE` - 开源协议

**命名规范**: 全大写，使用下划线分隔

### 2. 用户文档（User Documents）

**文档类型**:
- `USER_MANUAL.md` - 用户手册
- `QUICK_START.md` - 快速开始指南
- `FAQ.md` - 常见问题

**命名规范**: `USER_*` 前缀，全大写

### 3. 开发文档（Developer Documents）

**文档类型**:
- `DEVELOPER.md` - 开发者指南
- `API_REFERENCE.md` - API 参考
- `ARCHITECTURE.md` - 架构设计

**命名规范**: `DEV_*` 或 `DEVELOPER_*` 前缀

### 4. 构建文档（Build Documents）

**文档类型**:
- `*_BUILD_GUIDE.md` - 构建指南（如 `APK_BUILD_GUIDE.md`）
- `*_BUILD_QUICK.md` - 快速构建指南

**命名规范**: `*_BUILD_*` 格式

### 5. 功能文档（Feature Documents）

**文档类型**:
- `FEATURES.md` - 功能列表
- `*_FEATURE.md` - 特定功能说明（如 `REMINDER_FEATURE.md`）
- `*_PHASE*.md` - 功能阶段文档（如 `TASK_LOG_PHASE1.md`）

**命名规范**: `*_FEATURE` 或 `*_PHASE*` 格式

### 6. 测试文档（Testing Documents）

**文档类型**:
- `TESTING_GUIDE.md` - 测试指南
- `*_TEST_GUIDE.md` - 特定测试指南
- `TEST_CASES.md` - 测试用例

**命名规范**: `*_TEST*` 或 `TESTING_*` 格式

### 7. 审计文档（Audit Documents）

**文档类型**:
- `DOC_AUDIT_REPORT_V*.md` - 文档审计报告
- `DOC_ISSUES_LIST_V*.md` - 问题清单
- `DOC_MAINTENANCE_SUMMARY_V*.md` - 维护摘要

**命名规范**: `DOC_*_V<版本号>` 格式

### 8. 发布文档（Release Documents）

**文档类型**:
- `RELEASE_NOTES_v*.md` - 发布说明
- `*_DELIVERABLES_V*.md` - 交付物清单

**命名规范**: `RELEASE_*` 或 `*_DELIVERABLES_*` 格式

## 📝 文档命名规范

### 通用规则

1. **全大写**: 所有文档名使用全大写字母
2. **下划线分隔**: 单词之间使用下划线 `_` 分隔
3. **扩展名**: 统一使用 `.md`（Markdown 格式）
4. **版本号**: 使用 `_V<版本号>` 或 `_v<版本号>` 后缀

### 命名模式

```
<类型>_<功能>_<版本>.md

示例:
DOC_AUDIT_REPORT_V1.7.0_FINAL.md
USER_MANUAL_V1.7.0.md
APK_BUILD_GUIDE.md
TASK_LOG_PHASE1.md
```

### 特殊后缀

- `_FINAL` - 最终版本
- `_DRAFT` - 草稿版本
- `_QUICK` - 快速版本
- `_SUMMARY` - 摘要版本

## 📋 文档内容规范

### 文档结构

**标准结构**:
```markdown
# 文档标题

**版本**: v1.0  
**更新日期**: 2026-02-25  
**作者**: 开发团队

---

## 目录
- [章节1](#章节1)
- [章节2](#章节2)

---

## 章节1
内容...

## 章节2
内容...

---

**文档维护**: 开发团队  
**最后更新**: 2026-02-25
```

### 标题层级

- `#` - 文档标题（仅一个）
- `##` - 主要章节
- `###` - 次要章节
- `####` - 三级章节（尽量避免更深层级）

### 格式规范

1. **代码块**: 使用三个反引号 ` ``` ` 包裹，指定语言
2. **强调**: 使用 `**粗体**` 或 `*斜体*`
3. **列表**: 使用 `-` 或 `1.` 开头
4. **链接**: 使用 `[文字](URL)` 格式
5. **表格**: 使用 Markdown 表格语法

### 术语统一

**项目专用术语**:
- ✅ "任务执行日志" - 不使用"任务日志"
- ✅ "Bottom Sheet" - 不使用"抽屉"或"弹窗"
- ✅ "番茄钟" - 不使用"Pomodoro"
- ✅ "回收站" - 不使用"垃圾箱"

## 🔄 文档版本管理

### 版本号规则

**格式**: `v<主版本>.<次版本>.<修订版>`

- **主版本**: 重大功能变更（如 v1.0.0 → v2.0.0）
- **次版本**: 新功能添加（如 v1.6.0 → v1.7.0）
- **修订版**: Bug 修复或小改动（如 v1.7.0 → v1.7.1）

### 版本同步

**文档版本必须与代码版本同步**:
- `package.json` 更新版本号时，所有相关文档必须同步更新
- 版本号出现在文档中时，必须与 `package.json` 一致

### 版本归档

**归档策略**:
- 每个主版本发布后，将旧版本文档移至 `docs/releases/v<版本号>/` 目录
- 保留最近 3 个主版本的文档
- 更早的版本可以删除或压缩归档

## ✅ 文档审查流程

### 审查时机

1. **代码变更后**: 立即审查相关文档
2. **版本发布前**: 全面审查所有文档
3. **定期审查**: 每月进行一次文档审查

### 审查清单

**准确性检查**:
- [ ] 功能描述与代码实现一致
- [ ] 版本号与 package.json 一致
- [ ] 示例代码可以正常运行
- [ ] 截图/图片与当前版本一致

**完整性检查**:
- [ ] 所有新功能都有文档说明
- [ ] 所有 Bug 修复都记录在 CHANGELOG
- [ ] 所有 API 都有文档
- [ ] 所有配置项都有说明

**规范性检查**:
- [ ] 文档命名符合规范
- [ ] 文档结构符合模板
- [ ] 术语使用统一
- [ ] 格式符合 Markdown 规范

### 审查角色

- **开发人员**: 负责功能文档的准确性
- **测试人员**: 负责测试文档的完整性
- **技术写作**: 负责文档的规范性和可读性
- **外部审计员**: 负责全面审查（版本发布前）

## 🚫 文档禁忌

### 禁止行为

1. ❌ **夸大功能**: 不得描述代码中不存在的功能
2. ❌ **过时信息**: 不得保留已废弃功能的文档
3. ❌ **错误示例**: 不得提供无法运行的示例代码
4. ❌ **版本混乱**: 不得在文档中混用不同版本的信息
5. ❌ **术语不一**: 不得在不同文档中使用不同术语描述同一功能

### 警告行为

1. ⚠️ **延迟更新**: 代码变更后超过 24 小时未更新文档
2. ⚠️ **缺少审查**: 文档更新未经过审查直接合并
3. ⚠️ **格式混乱**: 文档格式不符合规范
4. ⚠️ **缺少版本**: 文档中未标注版本号
