# 任务关系类型说明

## 📊 本项目的4种任务关系

### 1️⃣ 前置任务（依赖关系 - 前向）
**字段**: `waitFor: Array<taskId>`  
**含义**: 当前任务需要等待的任务（必须先完成的任务）  
**视角**: 从当前任务看，哪些任务是我的前置条件

**示例**:
```
任务A: 完成需求文档
任务B: 开始编码 (waitFor: [任务A的ID])
```
- 对于任务B来说，任务A是它的**前置任务**
- 任务B必须等待任务A完成后才能开始

**特点**:
- 🔒 有前置任务的任务显示黄色"等待中"徽章
- 🚫 前置任务未完成时，当前任务无法标记为完成
- 🔗 支持多个前置任务：B.waitFor = [A1, A2, A3]

---

### 2️⃣ 后续任务（依赖关系 - 反向）
**字段**: 无独立字段，通过反向查询 `waitFor` 实现  
**含义**: 当前任务完成后才能开始的任务（等待我的任务）  
**视角**: 从当前任务看，哪些任务在等我完成

**示例**:
```
任务A: 完成需求文档
任务B: 开始编码 (waitFor: [任务A的ID])
```
- 对于任务A来说，任务B是它的**后续任务**
- 任务A完成后，任务B才能开始

**特点**:
- 🔓 有后续任务的任务显示"被依赖"状态
- ✅ 完成时自动通知所有后续任务可以开始
- 📊 任务详情页显示后续任务列表

---

### 3️⃣ 子任务（父子关系 - 纵向拆分）
**字段**: `parentTaskId: Number`  
**含义**: 从属于父任务的子任务，是父任务的组成部分  
**视角**: 大任务拆解为多个小任务，完成所有子任务 = 完成父任务

**示例**:
```
父任务: 完成项目报告
  ├─ 子任务1: 收集数据 (parentTaskId: 父任务ID)
  ├─ 子任务2: 制作图表 (parentTaskId: 父任务ID)
  ├─ 子任务3: 撰写报告 (parentTaskId: 父任务ID)
  └─ 子任务4: 团队审阅 (parentTaskId: 父任务ID)
```

**特点**:
- 👨‍👧 子任务记录父任务ID
- ✅ 完成父任务时自动完成所有子任务
- 🎯 由AI任务拆分功能自动生成
- 📊 任务详情页显示父子关系

---

### 4️⃣ 副任务（父子关系 - 横向并列）
**字段**: `parentTaskId: Number`（与子任务共用）  
**含义**: 与其他任务并列的任务，同属一个父任务，但彼此独立  
**视角**: 同一父任务下的多个子任务，可以并行执行

**示例**:
```
父任务: 完成项目报告
  ├─ 副任务1: 收集数据    ← 可以并行
  ├─ 副任务2: 制作图表    ← 可以并行
  ├─ 副任务3: 撰写报告    ← 可以并行
  └─ 副任务4: 团队审阅    ← 可以并行
```

**特点**:
- 🔄 副任务之间无依赖关系，可以并行执行
- 👥 多个副任务共享同一个父任务
- ⚡ 可以同时进行，互不阻塞
- 📊 在任务列表中平级显示

---

## 📋 四种关系对比表

| 关系类型 | 字段 | 方向 | 含义 | 视角 |
|---------|------|------|------|------|
| **前置任务** | `waitFor` | 前向依赖 | 我需要等待的任务 | 从当前任务看过去 |
| **后续任务** | 反向查询 | 后向依赖 | 等待我完成的任务 | 从当前任务看未来 |
| **子任务** | `parentTaskId` | 纵向拆分 | 父任务的组成部分 | 从父任务向下看 |
| **副任务** | `parentTaskId` | 横向并列 | 同一父任务下的并列任务 | 从兄弟任务横向看 |

---

## 🎯 关键区别

### 依赖关系（前置/后续）vs 父子关系（子任务/副任务）

#### 依赖关系：时间顺序
- **前置任务**: A完成 → B才能开始（时间先后）
- **后续任务**: A完成 → B可以开始（时间先后）
- **关系**: 控制执行顺序，强制串行
- **字段**: `waitFor` 数组

#### 父子关系：结构层级
- **子任务**: 父任务的组成部分（结构包含）
- **副任务**: 同一父任务下的并列任务（结构平级）
- **关系**: 表示任务拆解，可以并行
- **字段**: `parentTaskId` 单个ID

## 💡 实际使用场景

### 场景1：线性依赖（前置任务）
```
需求分析 → 设计方案 → 编码实现 → 测试验证
```
- 设计方案的前置任务：需求分析
- 编码实现的前置任务：设计方案
- 测试验证的前置任务：编码实现

### 场景2：任务拆解（子任务/副任务）
```
完成项目报告（父任务）
  ├─ 收集数据（副任务1）    ← 可以并行
  ├─ 制作图表（副任务2）    ← 可以并行
  ├─ 撰写报告（副任务3）    ← 可以并行
  └─ 团队审阅（副任务4）    ← 可以并行
```
- 4个副任务都是子任务
- 它们之间是副任务关系（并列）
- 可以同时进行，互不阻塞

### 场景3：复杂组合（依赖+拆解）
```
父任务A: 完成需求文档
  ├─ 副任务A1: 用户调研
  ├─ 副任务A2: 需求整理
  └─ 副任务A3: 文档撰写

父任务B: 开始编码
  ├─ 副任务B1: 前端开发
  ├─ 副任务B2: 后端开发
  └─ 副任务B3: 接口联调

依赖关系: B的前置任务是A
```
- A1、A2、A3是副任务（可并行）
- B1、B2、B3是副任务（可并行）
- 但B必须等A完成（依赖关系）

---

## 📊 总结

### 一句话区别

- **前置任务**: 我需要等谁完成（依赖关系，看过去）
- **后续任务**: 谁在等我完成（依赖关系，看未来）
- **子任务**: 我是谁的一部分（父子关系，纵向）
- **副任务**: 谁和我是兄弟（父子关系，横向）

### 实现方式

1. **前置任务 + 后续任务** = 依赖关系（`waitFor` 字段）
2. **子任务 + 副任务** = 父子关系（`parentTaskId` 字段）

### 核心理念

- **依赖关系**：控制时间顺序（串行）
- **父子关系**：表示结构层级（可并行）
